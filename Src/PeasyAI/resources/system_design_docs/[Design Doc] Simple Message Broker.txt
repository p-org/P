<title> Simple Message Broker (Kafka-like) </title>

<introduction>
This document describes the design of a simplified distributed message broker inspired by Apache Kafka. The system models the core mechanics of a publish-subscribe log: producers write keyed messages into an append-only topic log managed by a central broker, and consumers independently poll that log, each tracking their own read position. The broker is the single source of truth for the message log and for each consumer's committed offset.

The design intentionally constrains itself to a single topic with a single partition. This keeps the state space tractable for model checking while still exercising the interesting protocol interactions — in particular, the interplay between concurrent producers racing to append, multiple consumers progressing at different speeds, and the offset-commit protocol that ensures consumers can resume from a consistent position after a restart.

Key assumptions that bound the model:
  - There is exactly one topic backed by a single ordered partition; multi-partition fan-out is out of scope.
  - The network is reliable: every message sent between components is eventually delivered, and there are no duplicates or reorderings at the transport level.
  - Consumers read the log sequentially. Each consumer maintains a local cursor that advances through the log; random seeks are not supported.
  - The broker acknowledges every publish synchronously before the producer moves on, giving us a clear linearization point for each write.
  - Message delivery follows a pull model: consumers poll the broker at their own pace rather than having messages pushed to them, mirroring Kafka's real consumer protocol.
</introduction>

<components>
<source_components>

1. Broker
The broker is the heart of the system. It owns an append-only log — a sequence of (key, value) records, each stamped with a monotonically increasing offset starting at zero. When a producer publishes a message, the broker appends it to the tail of the log and replies with the offset that was assigned, serving as both an acknowledgment and a durable position marker.

On the consumer side, the broker maintains a registry of known consumers. A consumer must register before it can issue its first poll. Once registered, the broker tracks that consumer's last committed offset so that, if the consumer restarts, it can resume from where it left off. When a consumer polls, it supplies a starting offset; the broker slices the log from that point forward (up to a configurable batch size) and returns the batch. After the consumer has processed the batch, it sends an offset-commit request; the broker persists the new offset and acknowledges the commit, closing the read-process-commit cycle.

2. Producer
A producer is a straightforward client that publishes a fixed number of key-value messages to the broker, one at a time. After sending each publish request it blocks until it receives the broker's acknowledgment, ensuring that messages from a single producer are appended in the order they were sent. This synchronous send-and-wait pattern simplifies reasoning about ordering without sacrificing the ability to model concurrent writes from multiple producers.

</source_components>

<test_components>

3. Consumer
A consumer begins by registering with the broker and then enters a poll loop. On each iteration it asks the broker for the next batch of messages starting from its current offset, processes them in order, and commits the new offset back to the broker before polling again. The consumer validates that the offsets it receives are strictly increasing — a violation would indicate a bug in the broker's log management or batching logic. The loop terminates once the consumer has consumed a target number of messages.

<interactions>

1. ePublish
  Source: Producer → Broker
  Payload: producer (machine reference), key (int), value (int)

  A producer sends ePublish to write a single keyed message into the topic. The broker appends the record to the tail of its log, assigns the next sequential offset, and replies with ePublishAck. Because the producer waits for the ack before sending the next message, publishes from any one producer are totally ordered; however, interleaving across producers is non-deterministic and depends on scheduling.

2. ePublishAck
  Source: Broker → Producer
  Payload: offset (int)

  The broker's confirmation that the message has been durably appended. The offset tells the producer exactly where in the log its message landed. Upon receiving this event the producer is free to send its next message.

3. ePoll
  Source: Consumer → Broker
  Payload: consumer (machine reference), fromOffset (int)

  A consumer issues a poll to request the next slice of the log starting at fromOffset. The broker reads from that position up to its internal batch-size limit and responds with ePollResp. If fromOffset is already at or past the end of the log the broker returns an empty batch, and the consumer can retry later.

4. ePollResp
  Source: Broker → Consumer
  Payload: messages (sequence of (offset: int, key: int, value: int)), endOffset (int)

  The broker's response carrying a batch of log records. Each record includes its offset so the consumer can verify ordering. The endOffset field indicates the offset just past the last message in the batch, giving the consumer the value to use in its next poll or commit. After processing all messages in the batch the consumer issues an eCommitOffset.

5. eCommitOffset
  Source: Consumer → Broker
  Payload: consumer (machine reference), offset (int)

  After processing a batch, the consumer commits its new read position. This is the critical durability boundary: if the consumer were to crash and restart, it would resume from this committed offset rather than re-reading the entire log. The broker persists the offset in its consumer registry and acknowledges with eCommitOffsetAck.

6. eCommitOffsetAck
  Source: Broker → Consumer
  Payload: offset (int)

  Confirmation that the broker has recorded the consumer's committed offset. On receipt, the consumer proceeds to its next poll iteration.

7. eRegisterConsumer
  Source: Consumer → Broker
  Payload: consumer (machine reference)

  A one-time handshake that a consumer must complete before it can poll. The broker adds the consumer to its registry with an initial committed offset of zero, and responds with eRegisterConsumerAck. Attempting to poll without registering first would be a protocol violation.

8. eRegisterConsumerAck
  Source: Broker → Consumer
  Payload: (none)

  Acknowledgment that the consumer has been registered. The consumer transitions into its polling loop upon receiving this event.

</interactions>

<global_specifications>

1. OrderedDelivery (safety property)
Messages delivered to any consumer must arrive with strictly increasing offsets. Concretely, if a consumer observes offset `o` in one batch, every offset in any subsequent batch must be greater than `o`. A violation of this property would mean the broker is serving stale or out-of-order data, breaking the fundamental log abstraction.

2. NoMessageLoss (safety property)
Every message that a producer publishes and that the broker acknowledges must eventually be visible to consumers. Put differently, the broker's log must grow monotonically — an acknowledged offset must never disappear from the log, and no gaps may appear in the offset sequence.

3. ConsistentOffsets (safety property)
A consumer's committed offset must never exceed the broker's current log-end offset. If it did, the consumer would believe it had read messages that do not yet exist, which would corrupt its position tracking and potentially cause data loss on restart.

</global_specifications>

<possible_scenarios>

1. Single producer, single consumer, 5 messages — the happy-path baseline that exercises the full publish → poll → commit cycle end to end.
2. Two producers, single consumer, 3 messages each — tests that concurrent appends from different producers are correctly serialized in the log and that the consumer sees a consistent total order.
3. Single producer, two consumers, 5 messages — verifies that independent consumers can progress through the same log at different rates without interfering with each other's offsets.

</possible_scenarios>
